# PRD

## 1. 문서 목적

이 문서는 2026년 3월 기준 `codexia` 저장소의 실제 구현 상태를 바탕으로 제품 요구사항을 정리한다.
이 문서의 목적은 다음 두 가지다.

- 현재 제품이 이미 제공하는 기능 범위를 명확히 정의한다.
- 이후 기능 추가나 리팩터링 시 바뀌면 안 되는 핵심 사용자 가치와 운영 제약을 공유한다.

현재 제품은 저장소/패키지 이름으로는 `Codexia`를 사용하고, 웹 UI 브랜드 문자열은 `CodexOS`를 사용한다. 본 문서에서는 제품군 이름을 `Codexia`, UI 명칭은 필요 시 `CodexOS`로 구분한다.

## 2. 제품 정의

Codexia는 개발자용 AI 작업을 웹과 Telegram에서 실행하고 추적할 수 있는 멀티 에이전트 워크스페이스다.

핵심 개념은 단순 채팅이 아니라 다음 세 가지다.

- 세션 단위로 문맥을 보존한다.
- 작업을 `job` 단위로 분리해 생성, 실행, 실패, 완료를 추적한다.
- Codex CLI 또는 Gemini CLI 실행 결과를 실시간 스트리밍으로 보여준다.

## 3. 해결하려는 문제

기존 개발용 AI 인터페이스는 다음 문제를 자주 만든다.

- 작업 단위와 대화 단위가 섞여 현재 실행 상태를 추적하기 어렵다.
- 세션 문맥이 불안정해 이전 작업을 이어가기 어렵다.
- 에이전트가 실제 어떤 명령을 실행하는지 사용자 입장에서 관찰하기 어렵다.
- 로컬 워크스페이스 경계와 보호 경로 정책을 제품에 녹이기 어렵다.
- 웹과 메신저에서 같은 작업 맥락을 이어서 쓰기 어렵다.

Codexia는 이 문제를 세션 저장, job 스트리밍, trace 모드, 파일 참조 보조, Telegram 연동으로 해결한다.

## 4. 목표

### 제품 목표

- 개발자가 세션을 기준으로 AI 작업을 지속적으로 이어갈 수 있어야 한다.
- 응답 생성 과정을 job과 이벤트 기준으로 관찰할 수 있어야 한다.
- 웹 UI와 Telegram 모두에서 같은 백엔드 작업 계층을 사용할 수 있어야 한다.
- 로컬 파일 기반 저장소만으로도 MVP 수준의 운영이 가능해야 한다.
- 워크스페이스 보호 경로와 코어 경로 제약을 시스템 프롬프트에 반영해야 한다.

### 비목표

- 팀 단위 협업, 권한 관리, 조직용 멀티테넌트 지원
- 데이터베이스 기반 대규모 운영
- 브라우저 내 파일 편집기/IDE 기능
- 웹 사용자를 위한 로그인/인증 체계
- 분산 큐 또는 다중 워커 기반 백그라운드 실행

## 5. 대상 사용자

### 1차 사용자

- 로컬 저장소를 기준으로 AI 코딩 작업을 자주 반복하는 개인 개발자
- Codex CLI 또는 Gemini CLI를 UI에서 다루고 싶은 사용자
- 세션 이력과 실행 상태를 확인하면서 작업을 이어가고 싶은 사용자

### 2차 사용자

- Telegram에서 간단한 운영/개발 요청을 보내고 싶은 사용자
- 원격 환경에서 현재 job 상태나 최근 세션을 빠르게 확인하려는 사용자

## 6. 핵심 사용자 가치

- 세션을 잃지 않고 이어서 작업할 수 있다.
- 응답이 길어도 생성 과정을 실시간으로 볼 수 있다.
- trace 모드에서 reasoning, command, usage, TTFB를 확인할 수 있다.
- `@파일경로` 자동완성으로 파일 참조를 빠르게 넣을 수 있다.
- Telegram에서도 새 세션 생성, 기존 세션 전환, 작업 취소, 상태 조회가 가능하다.

## 7. 현재 범위

### 포함 범위

- 웹 메인 화면과 에이전트 워크스페이스
- 세션 생성, 조회, 삭제, 재진입
- 세션별 활성 job 1개 제한
- job 생성, 실행, SSE 스트리밍, 실패/취소/완료 처리
- Codex CLI 및 Gemini CLI 실행
- 모델 선택과 reasoning effort 선택
- trace 모드 기반 상세 이벤트 시각화
- JSON 파일 기반 세션/잡 저장
- Telegram 웹훅 처리 및 로컬 long polling 런타임
- Telegram 첨부 파일 다운로드 및 스크린샷 보조 기능
- 워크스페이스 루트, 보호 경로, 코어 경로 정책

### 제외 범위

- 사용자 계정 시스템
- 브라우저 업로드 기반 일반 파일 관리 UI
- 다중 작업 큐 우선순위 제어
- 장기 분석 리포트, 대시보드, 조직 단위 관리

## 8. 사용자 시나리오

### 시나리오 A: 웹에서 새 작업 시작

1. 사용자가 `/agent?new=1`로 진입한다.
2. 클라이언트가 새 `sessionId`를 생성한다.
3. 사용자가 모델, 사고수준, trace 모드를 선택하고 메시지를 입력한다.
4. `/api/agent/stream` 호출로 job이 생성된다.
5. 서버가 CLI를 실행하고 SSE로 생성 이벤트와 job 이벤트를 보낸다.
6. 사용자는 응답 텍스트, 명령 실행 상태, reasoning 로그를 실시간으로 본다.
7. 작업 완료 후 사용자/assistant 메시지가 세션 파일에 저장된다.

### 시나리오 B: 기존 세션 복구

1. 사용자가 메인 화면에서 기존 세션을 선택한다.
2. `/api/sessions/[sessionId]`가 세션과 활성 job 정보를 반환한다.
3. 활성 job이 있으면 클라이언트가 `/api/jobs/[jobId]/stream`으로 기존 스트림에 재연결한다.
4. 사용자는 이전 작업 이어보기와 새 작업 재개를 할 수 있다.

### 시나리오 C: Telegram에서 빠른 운영

1. 사용자가 Telegram으로 텍스트 또는 `/run` 명령을 보낸다.
2. 서버는 채팅 권한을 확인하고 세션을 결정한다.
3. 필요 시 첨부 파일을 로컬에 저장하고 프롬프트에 파일 경로를 덧붙인다.
4. 백엔드 job 계층으로 작업을 실행한다.
5. 결과를 Telegram 메시지, 상태 메시지, 이미지 응답으로 전달한다.

## 9. 기능 요구사항

### 9.1 웹 워크스페이스

- 사용자는 메인 화면에서 세션 목록을 볼 수 있어야 한다.
- 사용자는 세션을 다시 열고 이어서 작업할 수 있어야 한다.
- 사용자는 세션을 삭제할 수 있어야 한다.
- 사용자는 현재 활성 job이 있는 세션을 일괄 종료할 수 있어야 한다.
- 사용자는 워크스페이스 화면에서 메시지를 입력하고 스트리밍 응답을 확인할 수 있어야 한다.

### 9.2 세션 관리

- 세션은 `sessionId` 기반으로 식별되어야 한다.
- 세션에는 `messages`, `model`, `reasoningEffort`, `title`, `activeJobId`가 저장되어야 한다.
- 첫 응답 완료 시 제목이 비어 있으면 assistant 첫 줄 기반 자동 제목을 생성해야 한다.
- 세션은 로컬 JSON 파일로 저장되어야 한다.

### 9.3 Job 실행 및 상태 관리

- 새 요청마다 `jobId`를 생성해야 한다.
- 한 세션에는 동시에 하나의 활성 job만 허용해야 한다.
- job 상태는 `queued`, `running`, `completed`, `failed` 를 가져야 한다.
- job 이벤트는 chunk, reasoning, command, usage, metric, error를 누적 저장해야 한다.
- stale job은 heartbeat 기준으로 자동 실패 복구되어야 한다.
- 사용자는 활성 job을 취소할 수 있어야 한다.

### 9.4 스트리밍

- 웹은 SSE 기반으로 job 생성, snapshot, event, done, error 이벤트를 받아야 한다.
- 클라이언트는 기존 활성 job에도 재연결할 수 있어야 한다.
- trace 모드가 꺼져 있으면 텍스트 chunk 중심으로 응답을 보여줘야 한다.
- trace 모드가 켜져 있으면 reasoning, command, usage, ttfb_ms 이벤트를 반영해야 한다.

### 9.5 프롬프트 및 컨텍스트

- 시스템 프롬프트는 기본 프롬프트와 워크스페이스 정책을 합쳐 생성해야 한다.
- 대화 기록은 최근 메시지부터 예산에 맞게 잘라서 포함해야 한다.
- 입력 메시지는 최대 20,000자여야 한다.
- 긴 입력일수록 대화 예산을 줄이는 동적 컨텍스트 조절이 기본 활성 상태여야 한다.
- Telegram 채널에서는 전용 응답 형식 프롬프트를 추가로 삽입할 수 있어야 한다.

### 9.6 모델 실행

- 지원 모델은 현재 다음 다섯 가지여야 한다.
  - `gpt-5.4`
  - `gpt-5.3-codex`
  - `gpt-5.3-codex-spark`
  - `gemini-3.1-pro-preview`
  - `gemini-3-flash-preview`
- Codex 계열은 reasoning effort override를 적용해야 한다.
- Gemini 계열은 reasoning effort UI 선택을 비활성화하고 CLI 인자로는 사용하지 않아야 한다.
- CLI 실행 경로는 환경변수, PATH, 일반 설치 경로 fallback으로 해석해야 한다.

### 9.7 파일 참조 보조

- 사용자가 입력창에서 `@`를 입력하면 파일 자동완성을 제공해야 한다.
- 파일 검색은 워크스페이스 루트 기준 상대 경로를 반환해야 한다.
- 보호 경로와 `data`, `node_modules`, 숨김 디렉터리 등은 검색 대상에서 제외해야 한다.

### 9.8 Telegram 연동

- Telegram 웹훅 엔드포인트를 제공해야 한다.
- 별도 poller 런타임으로 Bot API `getUpdates` long polling을 지원해야 한다.
- 사용자는 `/run`, `/status`, `/cancel`, `/jobs`, `/new`, `/title`, `/model`, `/effort`, `/session`, `/recent`, `/screencap`, `/log`, `/clear`, `/ping`, `/id`, `/help` 계열 명령을 사용할 수 있어야 한다.
- 등록 코드 또는 허용 chat 목록을 통한 접근 제한을 지원해야 한다.
- 문서/사진 첨부를 로컬 파일로 저장하고 작업 프롬프트에 포함할 수 있어야 한다.

## 10. 비기능 요구사항

### 성능

- 응답은 가능한 한 빠르게 첫 바이트를 전송해야 하며, trace 모드에서는 `ttfb_ms`를 기록해야 한다.
- SSE polling 기반 job 스트림은 사용자 재접속 시 동일 job 이어보기를 지원해야 한다.

### 안정성

- 프로세스가 중간에 종료되더라도 stale job 복구가 가능해야 한다.
- 세션/잡 파일이 일부 손상되더라도 전체 서비스가 중단되지 않아야 한다.

### 보안/안전성

- 보호 경로는 사용자 요청이 있어도 읽기/검색/수정 대상에서 제외해야 한다.
- 코어 경로는 사용자가 명시적으로 허용한 경우에만 수정 대상으로 안내해야 한다.
- Telegram은 허용 chat 또는 등록 코드 기반 접근 제어를 지원해야 한다.

### 운영성

- 외부 DB 없이 로컬 파일만으로 동작해야 한다.
- 개발 모드에서 웹 서버와 Telegram poller를 함께 올릴 수 있어야 한다.

## 11. 성공 기준

- 웹에서 세션 생성 후 응답 스트리밍이 정상적으로 완료된다.
- 기존 세션 재진입 시 이력이 복구되고, 활성 job이 있으면 이어서 볼 수 있다.
- trace 모드에서 reasoning/command/usage 정보가 UI에 표시된다.
- Telegram에서 텍스트 실행, 상태 조회, 작업 취소가 가능하다.
- 보호 경로 설정 시 파일 자동완성과 프롬프트 정책에 반영된다.

## 12. 현재 운영 제약과 한계

- 저장소는 로컬 JSON 파일 저장소 기반이므로 다중 인스턴스 동기화에 적합하지 않다.
- job 취소와 실행 중 프로세스 추적은 현재 프로세스 메모리에 의존하므로 수평 확장 구조가 아니다.
- 웹 UI에는 별도 인증이 없으므로 배포 시 외부 접근 제어는 인프라 레벨에서 보완해야 한다.
- Telegram 기능은 Bot API, 로컬 브라우저/스크린샷 도구, 파일 시스템 접근 가능성에 영향을 받는다.

## 13. 향후 우선 후보

- 세션/잡 저장소를 SQLite 또는 별도 DB로 이전
- 웹 인증 및 사용자 권한 모델 추가
- job 검색/필터링/요약 뷰 강화
- trace 이벤트 시각화 개선
- 장기 보관 정책과 로그 정리 자동화
